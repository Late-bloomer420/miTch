1. Description of Processing

1.1 What We Do

miTch provides privacy-preserving identity verification. Users prove attributes about themselves (e.g., "I am over 18") without revealing the underlying personal data (e.g., their birthdate).

1.4 Processing Flow

Step 1 — Issuance: User authenticates with state provider (e.g., eID). Raw PII is received on-device, encrypted with an ephemeral key, predicates are derived locally, credential is signed, ephemeral key is destroyed. Raw PII is now mathematically irrecoverable.

Step 2 — Presentation: Verifier requests specific claims. User selects which claims to disclose via wallet consent screen. Only selected claims are sent as part of an SD-JWT selective disclosure proof.

Step 3 — Verification: Verifier validates the cryptographic proof. Receives only the disclosed boolean predicates. Cannot access any hidden claims. Issuer is not contacted and does not learn about the verification.

2. Necessity & Proportionality

2.1 Why This Processing Is Necessary

  • Legal compliance: EU regulations (DSA, Tobacco Products Directive, JuSchG) require age verification for certain services
  • Health sector: EHDS requires identity verification for health data access
  • Financial services: KYC regulations require identity verification

4. Measures to Address Risks

4.1 Technical Safeguards

🔐 Cryptographic

  • SD-JWT Selective Disclosure: Only chosen claims revealed. Verifier cannot access hidden claims.
  • Ed25519 Signatures: Issuer signs credentials, holder binds presentations to verifier nonce.
  • Crypto-Shredding: Ephemeral keys (AES-256) encrypt raw PII. Key destruction = provable data destruction.
  • StatusList2021 Revocation: Privacy-preserving bitstring. Verifier downloads entire list — issuer can't identify which credential was checked.

🛡️ Metadata Minimization

  • Response Padding: All responses padded to fixed 4KB — observers can't infer content from size.
  • Timing Jitter: 50-200ms random delay prevents timing correlation.
  • Disclosure Padding: Dummy disclosures added — hides actual number of claims.
  • Identical Decline Responses: "Declined" and "not available" produce identical responses.
  • Per-Session Identifiers: HMAC-derived keyIds — unlinkable across verifiers.

📋 Audit & Accountability

  • Tamper-Evident Audit Chain: Every action hash-chained. Any modification breaks the chain.
  • Consent Receipts: Every disclosure decision logged with evidence hashes (no PII in receipts).
  • ROPA (Art. 30): Aggregate processing records — counts only, no identifiers.
  • Crypto-Shred Proofs: Provable evidence that encryption keys were destroyed.

👤 User Control

  • Three-Tier Consent: Legal/Service/Optional claims clearly separated. Optional unchecked by default.
  • Equal-Weight Buttons: Approve and Decline are same size — no dark patterns.
  • Full Visibility: Users see exactly what is shared AND what is hidden.
  • Consent Revocation: Users can revoke any remembered consent at any time.
  • Data Portability: Export all consent receipts as JSON (Art. 20).

6. What miTch Structurally Cannot Do

These are not policy promises — they are architectural guarantees:

  • 🚫 Cannot store raw PII — ephemeral keys are destroyed after credential issuance
  • 🚫 Cannot track credential usage — issuer is never contacted during verification
  • 🚫 Cannot see what users share — selective disclosure happens on-device
  • 🚫 Cannot correlate users across verifiers — per-session derived identifiers
  • 🚫 Cannot recover shredded data — encryption key no longer exists
  • 🚫 Cannot distinguish declined from missing — identical responses by design

"miTch works not because it knows everything — but because it structurally cannot know anything."

7. Conclusion

This Data Protection Impact Assessment confirms that miTch's identity verification processing is compliant with GDPR Articles 5, 6, 7, 13, 15, 17, 20, 25, 30, 32, and 35.

The processing is necessary (legal requirements for identity verification), proportionate (only boolean predicates, not raw identity data), and safeguarded (crypto-shredding, selective disclosure, tamper-evident audit, metadata minimization).

Residual risks are low after mitigation. The architecture provides guarantees that exceed GDPR requirements — particularly for the right to erasure (Art. 17), where crypto-shredding provides mathematical proof of data destruction rather than relying on deletion promises.
