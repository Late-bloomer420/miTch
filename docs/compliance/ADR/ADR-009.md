# ADR-009: Native WebAuthn Verifier statt HMAC-Proxy-Mode-System

**Date:** 2026-03-14
**Status:** Accepted (retrospective — documents observed architectural divergence)
**Deciders:** Jonas (Late-bloomer420)

---

## Context

Zwischen Februar und März 2026 wurden zwei unterschiedliche Ansätze für WebAuthn-Verifikation spezifiziert bzw. implementiert:

**Spec-Ansatz (94, 95, 96):** Ein dreistufiges Mode-System `WEBAUTHN_VERIFY_MODE=allowlist|signed|native`, bei dem:
- `allowlist`: Assertions gegen eine statische Allowlist geprüft werden
- `signed` (Spec 94): Assertions als HMAC-SHA256 über `challenge|rpId|origin|issuedAt` verifiziert werden, Schlüssel `WEBAUTHN_ASSERTION_HMAC_SECRET`
- `native` (Spec 96): Adapter-Evidence als HMAC über `native|challenge|rpId|origin|issuedAt` verifiziert wird, Schlüssel `WEBAUTHN_NATIVE_ADAPTER_SECRET`

**Implementierter Ansatz:** Ein nativer WebAuthn-Verifier (`@mitch/webauthn-verifier`) ohne Mode-System, der direkt WebAuthn-Assertions verarbeitet:
- clientDataJSON parsen → challenge + origin prüfen
- authenticatorData parsen → counter auslesen
- Counter-Inkrement-Prüfung (Replay-Schutz)
- Single-use Challenge-Lifecycle (5 min Expiry, Löschung nach Verifikation)
- Step-up-auth Session-Management (Spec 67)

Zusätzlich existieren Folge-Specs (97-99, 101, 102), die auf dem Mode-System aufbauen und KPI-Felder, Config-Health-Guardrails und Runtime-Metriken für die drei Modi beschreiben.

## Problem

Der Code hat sich in eine architektonisch andere Richtung konkretisiert als die Specs 94/96 vorsahen. Die Specs sind nicht „falsch" — sie beschreiben einen validen Zwischenschritt. Aber das Repo hat diesen Zwischenschritt übersprungen und direkt einen nativen Verifier gebaut. Ohne explizite Dokumentation dieser Abweichung entsteht Verwirrung beim Lesen der Specs.

## Betrachtete Optionen

### Option A: HMAC-Proxy-Mode-System (Specs 94, 95, 96)

| Aspekt | Bewertung |
|--------|-----------|
| Env-basierte Konfiguration (`WEBAUTHN_VERIFY_MODE`) | Einfaches Deployment, Mode-Wechsel ohne Code-Änderung |
| HMAC-basierte Evidence | Kein echtes Authenticator-Parsing nötig |
| Drei Modi (allowlist → signed → native) | Gradueller Migrationspfad |
| Shared-Secret-Abhängigkeit | HMAC-Schlüssel muss sicher verteilt und rotiert werden |
| Kein Zugriff auf Authenticator-Counter | Replay-Schutz nur via Challenge-Einmaligkeit, nicht via Counter |
| Mode-Verwirrung möglich | Cross-mode Assertion-Confusion als Angriffsvektor (→ Spec 102) |

### Option B: Nativer WebAuthn-Verifier (implementiert)

| Aspekt | Bewertung |
|--------|-----------|
| Direkte Assertion-Verifikation | Standard-konform (W3C WebAuthn Level 2) |
| Counter-basierter Replay-Schutz | Zusätzliche Sicherheitsebene über Challenge-Einmaligkeit hinaus |
| Origin-Binding | Phishing-Schutz durch `clientData.origin`-Prüfung |
| Kein Shared Secret nötig | Kein HMAC-Key-Management |
| Kein Mode-System | Weniger Konfigurationskomplexität, aber auch kein gradueller Migrationspfad |
| `@simplewebauthn/server` als Dependency | Produktionsreife Signatur-Verifikation verfügbar (aktuell MVP-vereinfacht) |

## Entscheidung

Option B — nativer WebAuthn-Verifier ohne Mode-System.

Implementierung: `src/packages/webauthn-verifier/`

Zentrale Eigenschaften:
- `WebAuthnNativeVerifier` (index.ts): Challenge-Lifecycle, Authenticator-Registry, Assertion-Verifikation mit Counter-Replay-Schutz
- `StepUpAuthManager` (step-up-auth.ts, Spec 67): Session-Binding, Sensitivity-basierte Step-up-Trigger, Nonce-Binding
- `VerificationResult.reason`: 6 typisierte Deny-Gründe (`CHALLENGE_MISMATCH`, `CHALLENGE_EXPIRED`, `CHALLENGE_NOT_FOUND`, `COUNTER_REPLAY`, `SIGNATURE_INVALID`, `KEY_NOT_FOUND`)
- Integration mit Policy-Engine: `webauthn-reason-map.ts` mappt alle 6 Gründe auf `DenyReasonCode`

Tests: 8 Tests in `verifier.test.ts`, 9 Tests in `step-up-auth.test.ts` — alle grün.

## Begründung

1. **Sicherheitsniveau:** Nativer Verifier prüft Counter-Inkrement — das erkennt geklonte Authenticatoren, was HMAC nicht kann.
2. **Weniger Angriffsfläche:** Kein Shared-Secret-Management (HMAC-Keys), kein Mode-Confusion-Risiko.
3. **Standard-Nähe:** clientDataJSON + authenticatorData Parsing folgt der W3C WebAuthn Spec, nicht einem proprietären HMAC-Schema.
4. **Einfachere Architektur:** Ein Verifier statt drei Modi reduziert Konfigurationsfehler.

## Konsequenzen

**Positiv:**
- Stärkerer Replay-Schutz (Counter + Challenge-Einmaligkeit)
- Keine Shared-Secret-Rotation nötig
- Cross-mode Confusion architekturell ausgeschlossen (kein Mode-System → kein Risiko)
- `@simplewebauthn/server` bereits als Dependency vorhanden für künftige vollständige Signatur-Verifikation

**Negativ / Trade-offs:**
- Kein gradueller Migrationspfad (allowlist → signed → native) — der Verifier ist sofort „nativ"
- Specs 94, 96 beschreiben ein alternatives Modell, das nicht umgesetzt wurde — Leser müssen diese ADR kennen, um den Unterschied zu verstehen
- Signatur-Verifikation ist aktuell MVP-vereinfacht (clientData + counter, nicht volle COSE-Signatur) — Kommentar im Code: „in production use @simplewebauthn/server"

## Ausdrücklich nicht umgesetzt

| Spec-Konzept | Grund |
|-------------|-------|
| `WEBAUTHN_VERIFY_MODE` Env-Variable | Kein Mode-System nötig; nativer Verifier ist der einzige Pfad |
| `WEBAUTHN_ASSERTION_HMAC_SECRET` (Spec 94) | HMAC-Proxy ersetzt durch direkte Assertion-Verifikation |
| `WEBAUTHN_NATIVE_ADAPTER_SECRET` (Spec 96) | Kein Adapter-Evidence-HMAC; Verifier parst Assertions direkt |
| `WEBAUTHN_NATIVE_ADAPTER_OK` Flag (Spec 95) | Durch Spec 96 bereits als unsicher markiert; im nativen Verifier nicht relevant |
| Allowlist-Modus | Übersprungen zugunsten des nativen Ansatzes |

## Offene Folgepunkte

Die folgenden Specs bauen auf dem Mode-System auf und sind daher nur teilweise anwendbar. Ihre Kernanliegen (Observability, Config-Health) bleiben valide, müssten aber auf die tatsächliche Architektur adaptiert werden:

| Spec | Kernanliegen | Status | Möglicher Folge-Task |
|------|-------------|--------|----------------------|
| **97** (Config Health Guardrail) | Sichtbarkeit, ob WebAuthn korrekt konfiguriert ist | Die 3 geforderten KPI-Felder (`reauth_strong_enabled`, `verify_mode_code`, `secret_config_valid`) setzen Mode-System voraus; `webauthnDriftMs` in kpi.ts ist ein verwandtes aber anderes Konzept | Adaptiertes Config-Health-Feld für den nativen Verifier definieren (z.B. `webauthn_verifier_configured`, `rpId_set`) |
| **98** (Native Mode Visibility) | Sichtbarkeit, ob nativer Modus aktiv ist | Im aktuellen Design ist der Verifier immer nativ — das Feld `webauthn_native_mode_enabled` ist architekturell redundant | Kein Task nötig, solange kein Mode-System eingeführt wird |
| **99** (Allowlist Drift Warning) | Warnung bei Verbleib im schwachen Modus | Kein Allowlist-Modus vorhanden → Drift-Warning gegenstandslos | Kein Task nötig |
| **101** (Runtime Metrics) | Attempt/Success/Deny-Zähler für Pilot-Visibilität | Unabhängig vom Mode-System sinnvoll — der native Verifier könnte Attempts zählen | Sinnvoller Observability-Task für einen späteren Sprint |
| **102** (Cross-mode Edge Tests) | Schutz vor Assertion-Confusion zwischen Modi | Cross-mode Tests sind architekturell irrelevant (kein Mode-System); Challenge-Replay ist bereits getestet in `verifier.test.ts` | Kein Task nötig |
