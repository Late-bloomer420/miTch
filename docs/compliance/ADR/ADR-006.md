# ADR-006: StatusList2021 over OCSP for Revocation

**Date:** 2026-03-06
**Status:** Accepted
**Deciders:** Jonas (Late-bloomer420)

---

## Context

Credential revocation requires a mechanism for verifiers to check if a credential is still valid. Two dominant approaches:

| Mechanism | Privacy | Latency | Issuer Learns |
|-----------|---------|---------|---------------|
| OCSP (Online Certificate Status Protocol) | Low — issuer sees every check | Real-time | Which credential, when, from where |
| StatusList2021 (W3C) | High — verifier downloads entire list | Near-real-time (CDN) | Nothing (bulk download) |

## Decision

Use **StatusList2021** (W3C spec). The issuer publishes a compressed bitstring. Verifiers download the **entire list** via CDN and check locally. The issuer cannot determine which credential was checked.

Implementation: `src/packages/shared-crypto/src/status-list.ts` (G-02)

- Bitstring: GZIP-compressed, base64url-encoded
- CDN-hosted: CloudFront/S3, ~1KB per list (covers ~16K credentials)
- Fail-closed: if StatusList fetch fails → DENY (not ALLOW)

## Consequences

**Positive:** Issuer has zero knowledge of verification events. CDN-cacheable, low cost (~€0.0003/check). GDPR Art. 5(1)(c) compliant — no tracking by design.

**Negative:** Revocation propagation delay (CDN TTL, typically 5-15 min). List size grows with credential count (mitigated by compression). Status endpoint not yet deployed (🟡 in CIR matrix).
