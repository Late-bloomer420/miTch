# ADR-002: DPoP (RFC 9449) for Proof of Key Possession at Token Endpoints (E-05)

**Date:** 2026-03-06
**Status:** Accepted
**Deciders:** Jonas (Late-bloomer420)

---

## Context

CIR 2024/2979 requires that access tokens issued in the EUDI ecosystem be bound to the holder's key to prevent token theft and replay. Two mechanisms were evaluated:

| Mechanism | Standard | Sender-constrained | Replay protection | Notes |
|-----------|----------|-------------------|-------------------|-------|
| Bearer tokens | RFC 6750 | No | No | Vulnerable to theft; not acceptable |
| mTLS (RFC 8705) | RFC 8705 | Yes | Via TLS session | Requires PKI client certs; complex for mobile |
| DPoP (RFC 9449) | RFC 9449 | Yes | `jti` + `htm`/`htu` binding | EUDI ARF recommended; works over HTTPS |

---

## Decision

Adopt **DPoP (RFC 9449)** for proof of key possession at all token endpoints.

Implementation: `src/packages/shared-crypto/src/dpop.ts`

Key commitments:
- `typ: dpop+jwt` JOSE header (required by RFC 9449 §4.2)
- Embedded JWK in header (public key only — `d` stripped)
- `jti` (unique per proof), `htm` (HTTP method), `htu` (normalized URL), `iat` claims
- `nonce` claim when server-provided nonce is required (EUDI ecosystem uses nonces)
- `ath` claim (SHA-256 of access token) for token binding
- `normalizeHtu()` strips query string and fragment (RFC 9449 §6)
- Replay detection via caller-supplied `Set<string>` (seenJtis) — stateless callable

---

## Consequences

**Positive:**
- Sender-constrained tokens: even if an access token is stolen, it cannot be used without the DPoP private key
- No PKI client certificates required — works with ephemeral EC key pairs
- `jti` replay protection prevents proof reuse across requests
- `ath` binds proof to specific access token — prevents cross-token substitution
- RFC 9449 is adopted in HAIP and EUDI ARF as the mandatory binding mechanism

**Negative / Trade-offs:**
- Server must track `seenJtis` within a time window (1 hour) — requires shared state in distributed deployments (Redis / distributed set)
- `jti` Set implementation in `dpop.ts` is caller-managed — production issuers must wire their own persistent store
- Clock skew between wallet and server requires `iat` tolerance (currently ±300 s hardcoded)
