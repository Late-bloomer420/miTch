# ADR-004: Brainpool Curves (BSI/SOG-IS) + ECDH-HMAC MAC Verification (C-01, C-02)

**Date:** 2026-03-06
**Status:** Accepted (C-01 partial — P256r1 production-ready, P384r1 stub)
**Deciders:** Jonas (Late-bloomer420)

---

## Context

BSI TR-03116 and SOG-IS recommendations require Brainpool curves for qualified electronic signatures in the German/European regulated sector. Standard WebCrypto (SubtleCrypto) only supports NIST curves (P-256/P-384/P-521) — Brainpool is not in the WebCrypto spec. Additionally, BSI TR-03116 Annex B specifies ECDH + HMAC-SHA-256 MAC as an integrity mechanism for closed-ecosystem credential exchange.

---

## Decision

### C-01: Brainpool Curves

Use `@noble/curves` for pure-JS Brainpool curve implementation as a WebCrypto fallback.

Implementation: `src/packages/shared-crypto/src/brainpool.ts`

| Curve | Status | Notes |
|-------|--------|-------|
| `brainpoolP256r1` | ✅ Production-ready | RFC 5639 §3.4 parameters verified; `weierstrass()` + `ecdsa()` from `@noble/curves` |
| `brainpoolP384r1` | 🟡 Stub | RFC 5639 §3.6 parameters require BSI TR-03116 certified review before use; currently routes through P256r1 |

Pattern:
```typescript
const bp256Point = weierstrass(BP256_CURVE);
const bp256ECDSA = ecdsa(bp256Point, sha256);
```

Key exports: `generateBrainpoolKeyPair`, `signWithBrainpool`, `verifyWithBrainpool`, `brainpoolECDH`, `brainpoolPublicKeyToObject`

**Production note for qualified signatures:** Brainpool MUST be implemented via BSI-certified PKCS#11/HSM (e.g., SoftHSM2, Utimaco, Thales). The `@noble/curves` implementation is acceptable for testing and non-qualified use cases only.

### C-02: ECDH + HMAC-SHA-256 MAC

Use WebCrypto `deriveKey(ECDH)` to establish a shared HMAC key, then `crypto.subtle.sign/verify` for constant-time MAC computation.

Implementation: `src/packages/shared-crypto/src/mac-verify.ts`

- Key Agreement: P-256 ECDH (WebCrypto `deriveKey` → HMAC-SHA-256 key)
- MAC: `crypto.subtle.sign('HMAC', ...)` — constant-time, no custom comparison
- SD-JWT disclosure MAC: disclosures sorted alphabetically before concatenation with `~` — order-independent
- Use case: Closed-ecosystem issuer↔verifier flows where both parties share the ECDH key pair

---

## Consequences

**Positive:**
- BSI TR-03116 alignment: Brainpool P256r1 available for prototype/non-qualified use
- ECDH-HMAC MAC satisfies BSI TR-03116 Annex B integrity requirements for closed ecosystems
- `crypto.subtle.verify` provides constant-time MAC comparison — no timing side-channel
- `@noble/curves` is audited open-source; no custom field arithmetic
- Disclosure MAC is deterministic (sorted input) — replay of reordered disclosures detected

**Negative / Trade-offs:**
- `brainpoolP384r1` is a stub — BSI-verified 384-bit parameters not yet confirmed (RFC 5639 §3.6 parameter set must be independently verified against BSI TR-03111)
- No WebCrypto native Brainpool — `@noble/curves` runs in user-land JS, not in a secure element or HSM
- MAC-based verification is symmetric — verifier must hold the shared secret. Only suitable for pre-registered, closed ecosystems. Public verifiers MUST use ECDSA signatures instead.

### P384r1 Gap — Resolution Path

1. Obtain BSI TR-03116 / TR-03111 certified parameter set for `brainpoolP384r1` (RFC 5639 §3.6)
2. Verify parameters independently against RFC 5639 Appendix A test vectors
3. Replace stub implementation in `brainpool.ts` with verified `BP384_CURVE` constants
4. For qualified use: migrate to PKCS#11 via `@peculiar/webcrypto` + HSM driver
