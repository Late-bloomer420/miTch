# ADR-003: SIOPv2 + OID4VP + HAIP for Presentation Protocol (E-03, E-04, E-13)

**Date:** 2026-03-06
**Status:** Accepted
**Deciders:** Jonas (Late-bloomer420)

---

## Context

EUDI Wallet presentations require a standardized protocol for wallet-to-verifier interactions. CIR 2024/2982 mandates OID4VP as the primary presentation protocol and HAIP as the security profile for high-assurance use cases.

Three complementary specifications form the presentation stack:

| Spec | Purpose | Mandate |
|------|---------|---------|
| SIOPv2 | Self-Issued OpenID Provider — wallet acts as identity provider | CIR 2024/2982 |
| OID4VP | OpenID for Verifiable Presentations — credential presentation protocol | CIR 2024/2982; EUDI ARF |
| HAIP | High Assurance Interoperability Profile — security hardening on top of OID4VP | eIDAS high LoA |
| OAuth 2.0 Client Attestation | Wallet authenticity proof to verifier | CIR 2024/2979 |

---

## Decision

Adopt **SIOPv2 + OID4VP + HAIP** as the complete presentation protocol stack.

Implementations:
- `src/packages/oid4vp/src/siopv2.ts` — SIOPv2 request/response, pairwise sub
- `src/packages/oid4vp/src/client-attestation.ts` — OAuth 2.0 Client Attestation chain
- `src/packages/oid4vp/src/haip.ts` — HAIP Presentation Definition, Verifier Attestation, JWE response

### SIOPv2 (E-03)

- `response_type: id_token` — wallet issues a signed `id_token`
- `sub_jwk` embedded in `id_token` header — verifier can derive holder key without pre-registration
- Pairwise pseudonymous `sub`: `sha256base64url(verifierClientId + ":" + holderDID)` — prevents cross-verifier correlation
- Nonce and state binding enforced in `validateSIOPv2IDToken`

### OAuth 2.0 Client Attestation (E-04)

- Two-JWT chain: `oauth-client-attestation+jwt` (from attestation provider) + `oauth-client-attestation-pop+jwt` (from wallet)
- `cnf.jwk` in attestation binds wallet's public key
- PoP `jti` replay protection (caller-supplied `seenJtis`)
- Enables verifier to trust the wallet software without pre-registration

### HAIP (E-13)

- `client_id_scheme: verifier_attestation` — verifier must present a signed Verifier Attestation JWT
- `response_mode: direct_post.jwt` — presentation response sent as JWE-encrypted POST body
- `limit_disclosure=required` on all input descriptors — holder cannot over-disclose
- JWE response: `ECDH-ES+A256GCM` (CIR 2024/2982 mandated cipher suite)

---

## Consequences

**Positive:**
- Full EUDI ARF compliance for high-LoA presentation flows
- Pairwise sub prevents cross-verifier identity correlation (privacy-by-design)
- HAIP JWE response prevents network-level eavesdropping
- Client attestation allows verifier to distinguish genuine EUDI wallets from rogue apps
- `limit_disclosure=required` enforces data minimization (GDPR Art. 5(1)(c))

**Negative / Trade-offs:**
- Combined `vp_token`+`id_token` response path not yet wired in wallet-pwa (🟡 partial)
- Verifier Attestation requires issuance by a registered trust anchor — not yet integrated with EUDI Trust List
- HAIP adds complexity: verifiers must support JWE decryption on `direct_post.jwt` endpoint
