# ADR-008: Fail-Closed Policy Engine with Deny-Bias

**Date:** 2026-03-06
**Status:** Accepted
**Deciders:** Jonas (Late-bloomer420)

---

## Context

The policy engine decides whether to ALLOW, PROMPT, or DENY a verifier's request for credential attributes. The critical design question: what happens when the engine encounters ambiguity, missing data, or unexpected input?

| Strategy | On Ambiguity | Security | Usability |
|----------|-------------|----------|-----------|
| Fail-open (default ALLOW) | Data released | Low | High |
| Fail-closed (default DENY) | Data protected | High | Lower |

## Decision

The miTch policy engine is **fail-closed and deny-biased**. Any ambiguity, missing configuration, unknown verifier, expired credential, or unexpected input results in DENY.

Implementation: `src/packages/policy-engine/`

Key properties:
- 31 deny reason codes (G-03) — every denial is specific and auditable
- Anti-oracle design: 3-audience split (user gets full reason, verifier gets generic denial, audit gets hash)
- Deterministic evaluation: same input → same output, always
- No implicit trust: unknown verifiers are denied by default
- Critical bug found and fixed in G-02: StatusList2021 fail-closed was returning ALLOW on fetch failure

## Consequences

**Positive:** Security by default. Users never accidentally leak data. Every denial is traceable. Anti-oracle prevents verifiers from probing policy boundaries. GDPR Art. 25 compliance (data protection by default).

**Negative:** Stricter UX — legitimate verifiers must be properly configured. New use cases require explicit policy rules. More PROMPT dialogs for users (mitigated by auto-approve for minimal disclosures like single-predicate age checks).
