# 109 — Presentation Binding & Anti-Replay Spec v1

Stand: 2026-03-04
Status: Normative (P0 — blocks Phase 5 pilot)
Supersedes: Spec 20 (v0 draft) — this spec is the implementation-grade version.

---

## 1) Purpose

Define the complete binding and anti-replay mechanism for miTch presentation flows.
Prevents replay attacks, context-swap attacks, and audience mismatch exploits.

**Principle: Fail-closed. Any ambiguity = DENY.**

---

## 2) Canonicalization

### 2.1 Included Fields (Presentation Request Hash Input)

The following fields are extracted from a `PresentationRequest` and serialized in this exact order:

| # | Field Path           | Type     | Required |
|---|----------------------|----------|----------|
| 1 | `version`            | string   | yes      |
| 2 | `requestId`          | string   | yes      |
| 3 | `rp.id`              | string   | yes      |
| 4 | `rp.audience`        | string   | yes      |
| 5 | `purpose`            | string   | yes      |
| 6 | `claims`             | string[] | yes      |
| 7 | `policyRef`          | string   | no       |
| 8 | `binding.nonce`      | string   | yes      |
| 9 | `binding.expiresAt`  | string   | yes      |

`binding.requestHash` is **excluded** from hash input by definition.

### 2.2 Serialization Rules

1. Build a plain object with exactly the fields above (omit absent optional fields).
2. Serialize using `canonicalStringify()` from `@mitch/shared-crypto`:
   - UTF-8 encoding
   - Lexicographic key ordering (recursive)
   - No insignificant whitespace
   - NFC string normalization
   - Arrays preserve sender order
3. Hash with SHA-256.
4. Encode as **lowercase hex** (consistent with existing `sha256()` in shared-crypto).

### 2.3 Determinism Guarantee

Two conforming implementations given identical input MUST produce identical hash output.
Test vectors are provided in `@mitch/shared-crypto` test suite.

## 3) Nonce Lifecycle

### 3.1 Generation

- 32 cryptographically random bytes, encoded as hex (64 chars).
- Generated by the **verifier** when creating a presentation request.

### 3.2 Storage

- Key: `(audience, nonce)` composite.
- TTL: configurable, default **5 minutes**.
- Maximum store size: configurable, default 100,000 entries.
- Eviction: expired entries pruned on access + periodic cleanup.

### 3.3 Consumption

- **Single-use**: a nonce can be consumed exactly once.
- `consume(audience, nonce, now)` → `{ ok: true }` or `{ ok: false, code: string }`.
- Atomic: check + delete in one operation (no TOCTOU).
- After consumption, the nonce is **removed** from the store.

### 3.4 Failure Codes

| Condition                    | Deny Code                        |
|------------------------------|----------------------------------|
| Nonce not found              | `DENY_BINDING_NONCE_UNKNOWN`     |
| Nonce already consumed       | `DENY_BINDING_NONCE_REPLAY`      |
| Nonce expired (beyond skew)  | `DENY_BINDING_EXPIRED`           |
| Audience mismatch            | `DENY_BINDING_AUDIENCE_MISMATCH` |
| Hash mismatch                | `DENY_BINDING_HASH_MISMATCH`     |
| Missing required field       | `DENY_SCHEMA_MISSING_FIELD`      |

---

## 4) Clock Skew

- Tolerance: configurable, default **±30 seconds**.
- Applied to `binding.expiresAt` check:
  `expiresAt + clockSkewMs >= now` → not expired.
- Skew beyond tolerance: `DENY_BINDING_EXPIRED`.

---

## 5) Binding Validation Order

The verifier MUST validate in this exact order (short-circuit on first failure):

1. All required fields present → else `DENY_SCHEMA_MISSING_FIELD`
2. `binding.nonce` exists in nonce store for `rp.audience` → else `DENY_BINDING_NONCE_UNKNOWN`
3. `binding.expiresAt` not expired (with skew) → else `DENY_BINDING_EXPIRED`
4. `rp.audience` matches verifier's own identifier → else `DENY_BINDING_AUDIENCE_MISMATCH`
5. Recompute request hash, compare to `binding.requestHash` → else `DENY_BINDING_HASH_MISMATCH`
6. Consume nonce atomically → else `DENY_BINDING_NONCE_REPLAY`

---

## 6) Security Notes

- No global correlation IDs in canonicalized fields.
- Nonce store MUST NOT persist across process restarts in production without encryption.
- Hash algorithm agility: `version` field enables future SHA-3 migration.
- Rate limiting on `/present` is orthogonal but recommended (already implemented).
