### F-01 · Kryptographie · HOCH
**Recovery ist kein Shamir — sondern 3-of-3 XOR**

- **Datei:** `src/packages/shared-crypto/src/recovery.ts`
- **Problem:** Kommentar behauptet "Shamir's Secret Sharing (2-of-3)", aber die Implementierung ist ein XOR-Schema: `F1 ⊕ F2 ⊕ F3 = Key`. Alle drei Fragmente sind nötig. `recover()` wirft bei `fragments.length < 3`. Ein verlorenes Fragment = Totalverlust.
- **Impact:** Recovery-Versprechen an User/Stakeholder ist falsch. Für eine Trust-Circle-Lösung mit Vertrauenspersonen ist 3-of-3 inakzeptabel.
- **Fix:** Entweder `sss-wasm` integrieren (bereits als TODO in `ADVANCED_SECURITY_HARDENING.ts:95` notiert) oder Dokumentation + API auf 3-of-3 korrigieren.
- **Aufwand:** M

### F-02 · Sicherheit · HOCH
**`matchesPattern` ist ReDoS-anfällig**

- **Datei:** `src/packages/policy-engine/src/engine.ts:489-494`
- **Problem:** `new RegExp('^' + pattern.replace(/\*/g, '.*') + '$')` — Wenn ein Pattern Regex-Sonderzeichen enthält (`(`, `+`, `{`, etc.), entsteht ein kaputter oder exponentiell langsamer Regex. Im Override-Fall (Zeile 186) wird `request.verifierId` als Pattern gesetzt — und das kommt vom Verifier (untrusted Input).
- **Impact:** Denial-of-Service gegen die Policy Engine. Ein bösartiger Verifier könnte mit einem crafted `verifierId` die Evaluation blockieren.
- **Fix:** Sonderzeichen escapen vor Regex-Konstruktion, oder Pattern-Matching durch einfaches String-Splitting ersetzen (Glob → Split auf `*`).
- **Aufwand:** S

### F-03 · Sicherheit · MITTEL
**Request-Hash im DecisionCapsule ist kein echter Hash**

- **Datei:** `src/packages/policy-engine/src/engine.ts:606-607`
- **Problem:** `sha256(req:${request.verifierId})` ist ein String-Literal, kein tatsächlicher SHA-256-Hash. Ebenso `policyHash`. Das DecisionCapsule ist damit nicht integritätsgesichert gegen Manipulation der Request-Daten.
- **Impact:** Ein Angreifer, der das Capsule abfängt, kann nicht verifizieren, ob der Request unverändert war. Die Wallet-Attestation-Signatur (Zeile 697) schützt zwar den Capsule-Inhalt, aber der Hash selbst ist wertlos.
- **Fix:** Echten SHA-256 über `canonicalStringify(request)` und `canonicalStringify(policy)` berechnen.
- **Aufwand:** S

### F-04 · Architektur · MITTEL
**Drei verschiedene EphemeralKey-Klassen**

- **Dateien:**
  - `src/packages/shared-crypto/src/ephemeral.ts` (CryptoKey-basiert, WebCrypto)
  - `src/packages/shared-crypto/src/ephemeral-key.ts` (Uint8Array-basiert, 34 Zeilen, minimalistisch)
  - `src/packages/secure-memory/src/ephemeral_key.ts` (Node-only, Hash-Proof, Timeout)
- **Problem:** Drei Klassen mit überlappender Verantwortung. `pairwise-did.ts` importiert die einfache Klasse (keine Shred-Proofs). `ephemeral.ts` setzt `shred()` nur via Reference-Drop (`this.key = null`), kein auditfähiger Nachweis.
- **Impact:** Inkonsistentes Shredding-Verhalten. Audit-Proofs nur in einer der drei Varianten. Verwirrung für Contributors.
- **Fix:** Ein Interface `IEphemeralKey` definieren, eine primäre Implementierung pro Umgebung (Browser vs. Node), andere als Adapter.
- **Aufwand:** M
- **Abhängigkeiten:** Beeinflusst `pairwise-did.ts`, `ephemeral.ts`, `WalletService.ts`

### F-05 · Sicherheit · MITTEL
**EphemeralKey ist `extractable: true`**

- **Datei:** `src/packages/shared-crypto/src/ephemeral.ts:44`
- **Problem:** `true, ['encrypt', 'decrypt']` — das ist bewusst (Key-Wrapping), aber ein Angreifer mit Heap-Zugriff könnte `exportKey('raw', ...)` vor dem Shred aufrufen.
- **Impact:** Im Browser-Kontext ohne Secure Enclave ist das Key-Material prinzipiell extrahierbar. Für PoC akzeptabel, für Production nicht.
- **Fix:** Kurzfristig dokumentieren als Known Limitation. Langfristig: T-31 TEE-Migration abschließen, Key-Wrapping über non-extractable Keys lösen.
- **Aufwand:** L (architekturelle Änderung)

### F-06 · Korrektheit · NIEDRIG
**Pairwise-DID HKDF: Keine Validierung des abgeleiteten P-256 Private Key**

- **Datei:** `src/packages/shared-crypto/src/pairwise-did.ts` (Funktion `generatePairwiseDIDFromMasterKey`)
- **Problem:** 32 HKDF-Bytes werden direkt als P-256 Private Key verwendet. Nicht jeder 32-Byte-Wert ist ein gültiger P-256-Skalar (muss < n sein). Wahrscheinlichkeit eines Fehlers: ~1 in 2^128, aber in einem sicherheitskritischen System sollte `importKey` mit Error-Handling abgesichert sein.
- **Fix:** Try/Catch um `crypto.subtle.importKey('pkcs8', ...)` mit Retry-Logik.
- **Aufwand:** S

### F-07 · Architektur · MITTEL
**Selective Decryption ist Pseudo-Minimierung**

- **Datei:** `src/packages/secure-storage/src/index.ts:197-209`
- **Problem:** `loadSelectiveClaims()` entschlüsselt den gesamten Blob und filtert danach. Alle Claims sind kurzzeitig im RAM, auch nicht-angeforderte.
- **Impact:** Widerspricht dem Claim "Minimize before decrypt" — es ist "decrypt, then filter".
- **Fix:** Kurzfristig: Dokumentation anpassen ("post-decryption minimization"). Langfristig: Claim-Level-Encryption (jeder Claim ein eigener Ciphertext-Blob).
- **Aufwand:** L

### F-08 · Sicherheit · NIEDRIG
**`getRawDocument()` ist public ohne Build-Guard**

- **Datei:** `src/packages/secure-storage/src/index.ts:266-276`
- **Problem:** Test-only Methode, die verschlüsselte Dokumente ohne Entschlüsselung zurückgibt. Kein `@internal`, kein Build-Flag.
- **Impact:** In Production könnte ein Integrator versehentlich Ciphertext exponieren.
- **Fix:** `@internal` Tag + separate Test-Export-Map in `package.json`, oder hinter `process.env.NODE_ENV === 'test'` Guard.
- **Aufwand:** S

### F-09 · Architektur · MITTEL
**Strict Verifier Binding ist komplett auskommentiert**

- **Datei:** `src/packages/policy-engine/src/engine.ts:460-470`
- **Problem:** Der gesamte Origin-Check-Block ist Kommentar. Ein Verifier kann sich als `did:mitch:verifier-liquor-store` ausgeben, aber von `evil.com` operieren.
- **Impact:** Verifier-Spoofing ist trivial. `strictVerifierBinding: false` in DemoPolicy macht das explizit, aber es fehlt ein konkreter Implementierungsplan.
- **Fix:** Phase 1: Einfacher Origin-vs-VerifierID-Check. Phase 2: DNS-DID-Binding via `.well-known/did-configuration`.
- **Aufwand:** M (Phase 1), L (Phase 2)

### F-10 · Infrastruktur · HOCH

- **Dateien:**

  - `.github/workflows/ci.yml` triggert auf `main`, Default-Branch ist `master`

- **Fix:** Branch-Namen vereinheitlichen, `pnpm` durchgängig verwenden, Tests re-enablen, `continue-on-error` entfernen.
- **Aufwand:** S
